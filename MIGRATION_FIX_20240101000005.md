# Migration Fix: 20240101000005_add_dashboard_indexes.sql

## Issue Found

Migration `20240101000005_add_dashboard_indexes.sql` failed with error:
```
ERROR: 42703: column "status" does not exist
```

## Root Cause

The migration incorrectly referenced columns that don't exist in the assessments table:

### Incorrect Code
```sql
CREATE INDEX IF NOT EXISTS idx_assessments_org_status 
ON assessments(organization_id, status)
WHERE status = 'in_progress';
```

### Problems
1. ❌ `assessments` table doesn't have `organization_id` column
   - Organization filtering is done via `participant_id` join to `participants` table
   
2. ❌ `assessments` table doesn't have `status` column
   - Uses `is_complete` (boolean) instead of `status` (text)

## Fix Applied

### Corrected Code
```sql
-- Index for "Pending assessments" query
-- Covers: participant_id + is_complete filter
-- Note: assessments table doesn't have organization_id - filter via participants join
CREATE INDEX IF NOT EXISTS idx_assessments_participant_incomplete 
ON assessments(participant_id, is_complete)
WHERE is_complete = FALSE;

-- Index for assessments by completion date
CREATE INDEX IF NOT EXISTS idx_assessments_participant_completed 
ON assessments(participant_id, completed_at DESC)
WHERE is_complete = TRUE;
```

### Changes
1. ✅ Changed `organization_id` to `participant_id` (correct column)
2. ✅ Changed `status` to `is_complete` (correct column)
3. ✅ Changed `WHERE status = 'in_progress'` to `WHERE is_complete = FALSE`
4. ✅ Added second index for completed assessments
5. ✅ Updated documentation to clarify organization filtering via join

## Assessments Table Schema (Actual)

From `20240101000002_create_intake_assessment_tables.sql`:

```sql
CREATE TABLE assessments (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  participant_id UUID REFERENCES participants(id) ON DELETE CASCADE NOT NULL,
  assessment_type TEXT NOT NULL CHECK (assessment_type IN ('SUPRT_C', 'BARC_10', 'SSM')),
  
  -- Session tracking
  started_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  completed_at TIMESTAMP WITH TIME ZONE,
  is_complete BOOLEAN DEFAULT FALSE,  -- ✅ This is the correct column
  
  -- Scores and data
  total_score INTEGER,
  item_scores JSONB,
  responses JSONB,
  conversation_transcript TEXT,
  
  -- Metadata
  conducted_by UUID REFERENCES users(id),
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

**Key Points:**
- ✅ Has `participant_id` (not `organization_id`)
- ✅ Has `is_complete` (not `status`)
- ✅ Organization filtering requires join: `assessments → participants → organization_id`

## Impact on Dashboard Queries

The corrected indexes will optimize these Dashboard queries:

### Pending Assessments Count
```sql
-- Before: Full table scan
-- After: Uses idx_assessments_participant_incomplete
SELECT COUNT(*) 
FROM assessments a
INNER JOIN participants p ON a.participant_id = p.id
WHERE p.organization_id = $1 
  AND a.is_complete = FALSE;
```

### Recent Completed Assessments
```sql
-- Before: Full table scan + sort
-- After: Uses idx_assessments_participant_completed
SELECT a.* 
FROM assessments a
INNER JOIN participants p ON a.participant_id = p.id
WHERE p.organization_id = $1 
  AND a.is_complete = TRUE
ORDER BY a.completed_at DESC
LIMIT 10;
```

## Lessons Learned

1. **Always verify column names** against actual schema before creating indexes
2. **Check for organization_id** - not all tables have it directly
3. **Understand join patterns** - some tables require joins for org filtering
4. **Test migrations** on staging before production
5. **Use IF NOT EXISTS** for idempotent migrations
